## Learnings

**WKWebView is broken in this app:** Spawns a sandboxed WebContent process that can't access system services, producing floods of sandbox errors. Do not use.

**NSHostingController + NSViewRepresentable causes ViewBridge errors:** SwiftUI-AppKit bridging overhead for document-style windows causes "ViewBridge to RemoteViewService Terminated" errors. Bypass SwiftUI entirely for these windows.

**Working pattern for help/document windows:** Build the NSScrollView directly in AppDelegate and set `win.contentView = scrollView`. No view controller, no NSHostingController needed.

**Don't use `showHelp(_:)` as a selector:** NSApplication intercepts this and shows "Help isn't available for backit" system alert. Use any other name (e.g. `openHelpWindow(_:)`).

**Settings menu:** `Settings { EmptyView() }` is required to satisfy SwiftUI `@main` App protocol, but it adds a "Settings…" menu item. Suppress with `.commands { CommandGroup(replacing: .appSettings) { } }`.

**HelpContent.swift maintenance:** `docs/user-manual.md` is the source of truth. If the manual is updated, `HelpContent.swift` must be manually regenerated (HTML conversion). There's no automated pipeline for this yet.

## Post-Mortem

### What Worked
- `NSAttributedString(html:)` rendered in-process via NSTextView — clean, no sandbox issues
- `CommandGroup(replacing: .help)` with a custom selector for the Help menu
- `PBXFileSystemSynchronizedRootGroup` — new files in the `backit/UI/` directory are auto-included in the build; no xcodeproj editing needed

### What Failed
- WKWebView → WebContent process sandbox errors, broken window
- NSHostingController(rootView: HelpView()) → ViewBridge/RemoteViewService errors
- `showHelp(_:)` selector → intercepted by NSApplication, shows system alert

### Key Decisions
- **HTML embedded in Swift string (HelpContent.swift)** rather than bundling user-manual.md as a resource — avoids xcodeproj resource phase complexity; tradeoff is manual sync if docs change
- **NSTextView direct in AppDelegate** — simplest working approach after two failed alternatives
- **HelpView.swift left in tree** — unused NSViewRepresentable stub; harmless since it's never instantiated
